# 품질 체크리스트 (Quality Criteria)

학습 모듈 **생성 시점**에 적용하는 4가지 핵심 기준입니다.
Phase 3 (설계 + 품질 검증)에서 Sequential Thinking 4회 동안 검증합니다.

---

## 사용 시점

```
Phase 3: 설계 + 품질 검증
├── 9회: 정확성 검증 ← 기준 1
├── 10회: 교육적 구조 검증 ← 기준 2
├── 11회: 학습자 친화성 검증 ← 기준 3
└── 12회: 실용성 검증 ← 기준 4
```

## 기준 1: 정확성 (Accuracy)

### 정의
기술적 설명과 코드 예제가 정확하고 최신 정보를 반영하는가?

### 체크리스트

| 항목 | 확인 질문 |
|------|----------|
| 공식 문서 일치 | 2025년 Android 공식 문서와 내용이 일치하는가? |
| API 정확성 | 함수 시그니처, 파라미터, 반환값이 정확한가? |
| deprecated 체크 | 사용된 API가 deprecated되지 않았는가? |
| 코드 동작 | 예제 코드가 실제로 컴파일되고 동작하는가? |
| 버전 호환성 | Compose 최신 버전과 호환되는가? |

### 검증 방법

```kotlin
// 1. 빌드 검증
./gradlew :study:{category}:{module}:compileDebugKotlin

// 2. 공식 문서 교차 검증
WebSearch: "{주제} Android Compose 2025 official"
Context7: get-library-docs(topic="{주제}")
```

### 등급 기준

| 등급 | 기준 |
|------|------|
| ✅ 우수 | 모든 항목 통과, 공식 문서와 100% 일치 |
| ⚠️ 양호 | 1-2개 경미한 오류, 수정 가능 |
| ❌ 미흡 | deprecated API 사용 또는 심각한 오류 |

## 기준 2: 교육적 구조 (Pedagogical Structure)

### 정의
주제의 특성에 맞는 적절한 교육 틀을 사용하고 있는가?

### 체크리스트

| 항목 | 확인 질문 |
|------|----------|
| 틀 적합성 | 주제 유형에 맞는 패턴(A/B/C/D)을 사용하는가? |
| 논리적 흐름 | 내용이 자연스럽게 연결되는가? |
| 단계적 진행 | 쉬운 것에서 어려운 것으로 진행하는가? |
| 완결성 | 주제를 충분히 다루고 있는가? |

### 틀 적합성 판단

```
주제 유형 → 적합한 틀

Side Effect (LaunchedEffect, DisposableEffect)
  → 패턴 A: 문제-해결

필수 구성 요소 (Modifier, Text, Button)
  → 패턴 B: 구성요소 탐구

선택적 옵션들 (Layout, Navigation)
  → 패턴 C: 비교-선택

설계/아키텍처 (ViewModel, State hoisting)
  → 패턴 D: 아키텍처 가이드
```

### 잘못된 틀 예시

```
❌ 잘못된 예:
주제: Modifier
틀: "Modifier를 사용하지 않으면 발생하는 문제!"

→ Modifier는 Compose 필수 구성 요소이므로
  "없으면 문제" 틀이 아닌 "구성요소 탐구" 틀이 적합

✅ 올바른 예:
주제: Modifier
틀: "Modifier의 다양한 기능과 조합 방법"
```

### 등급 기준

| 등급 | 기준 |
|------|------|
| ✅ 우수 | 주제에 최적인 틀 사용, 논리적 흐름 완벽 |
| ⚠️ 양호 | 틀은 적합하나 흐름에 개선 여지 |
| ❌ 미흡 | 주제에 부적합한 틀 사용 |

## 기준 3: 학습자 친화성 (Learner Friendliness)

### 정의
Compose 입문자가 이해하기 쉽게 작성되었는가?

### 체크리스트

| 항목 | 확인 질문 |
|------|----------|
| 용어 설명 | 전문 용어에 대한 충분한 설명이 있는가? |
| 배경 지식 | 필요한 사전 지식을 명시했는가? |
| 시각적 자료 | 다이어그램, 예시 화면이 충분한가? |
| 단계별 안내 | 코드를 단계별로 설명하는가? |
| 비유/예시 | 이해를 돕는 비유나 현실 세계 예시가 있는가? |

### 언어 수준 가이드

```markdown
❌ 피해야 할 표현:
"Recomposition이 트리거되면 해당 Composable의 스코프 내에서..."
"LaunchedEffect의 블록은 코루틴 스코프에서 실행되어..."

✅ 권장 표현:
"화면이 다시 그려질 때(Recomposition), 해당 UI 요소 안에서..."
"LaunchedEffect 안의 코드는 비동기로 실행됩니다 (백그라운드 작업처럼)..."
```

### 비유 예시

```markdown
✅ 좋은 비유:
"LaunchedEffect는 '일꾼'과 같습니다.
 - UI가 나타나면 일꾼이 일을 시작합니다
 - key가 바뀌면 기존 일꾼은 해고되고 새 일꾼이 옵니다
 - UI가 사라지면 일꾼도 퇴근합니다"

✅ 좋은 비유:
"remember는 '메모장'과 같습니다.
 - 화면이 다시 그려져도 메모장의 내용은 유지됩니다
 - 새 화면으로 이동하면 메모장을 버립니다"
```

### 등급 기준

| 등급 | 기준 |
|------|------|
| ✅ 우수 | Compose 초보자도 완전히 이해 가능 |
| ⚠️ 양호 | 대부분 이해 가능, 일부 설명 보완 필요 |
| ❌ 미흡 | 전문 용어 남발, 배경 지식 없으면 이해 불가 |

## 기준 4: 실용성 (Practicality)

### 정의
실제 앱 개발에 바로 적용할 수 있는 예제와 연습을 제공하는가?

### 체크리스트

| 항목 | 확인 질문 |
|------|----------|
| 현실적 예제 | 실제 앱에서 사용되는 시나리오인가? |
| 복사-붙여넣기 | 코드를 바로 사용할 수 있는가? |
| 응용 가이드 | 다른 상황에 어떻게 응용하는지 설명하는가? |
| 연습 적절성 | 연습 문제가 학습 목표와 일치하는가? |
| 난이도 균형 | 쉬운/중간/어려운 문제가 균형있는가? |

### 예제 품질 기준

```kotlin
❌ 비현실적 예제:
@Composable
fun Example() {
    LaunchedEffect(Unit) {
        println("Hello")  // 실제 앱에서 이렇게 안 씀
    }
}

✅ 현실적 예제:
@Composable
fun UserProfileScreen(userId: String) {
    var user by remember { mutableStateOf<User?>(null) }
    var isLoading by remember { mutableStateOf(true) }

    LaunchedEffect(userId) {
        isLoading = true
        user = repository.fetchUser(userId)  // 실제 데이터 로딩
        isLoading = false
    }

    when {
        isLoading -> LoadingIndicator()
        user != null -> UserProfile(user!!)
        else -> ErrorMessage()
    }
}
```

### 연습문제 난이도 균형

```markdown
✅ 균형 잡힌 연습문제:
연습 1: 기본 API 호출 (쉬움)
  - 단일 개념만 사용
  - 힌트가 구체적

연습 2: 두 개념 조합 (중간)
  - 이 모듈 + 이전 모듈 개념 조합
  - 힌트가 방향만 제시

연습 3: 실제 앱 시나리오 (어려움)
  - 여러 개념 통합
  - 힌트가 최소화
  - 실무에서 만날 수 있는 상황
```

### 등급 기준

| 등급 | 기준 |
|------|------|
| ✅ 우수 | 바로 실무에 적용 가능한 예제, 균형 잡힌 연습 |
| ⚠️ 양호 | 대부분 실용적, 일부 예제 개선 필요 |
| ❌ 미흡 | 너무 단순하거나 비현실적인 예제 |

## 종합 등급 산정

### A+ 등급 조건

```
□ 정확성: ✅ 우수
□ 교육적 구조: ✅ 우수
□ 학습자 친화성: ✅ 우수
□ 실용성: ✅ 우수

→ 4가지 모두 ✅면 A+ 등급!
```

### 등급 테이블

| 최종 등급 | 조건 |
|----------|------|
| **A+** | 4가지 기준 모두 ✅ 우수 |
| **A** | 3가지 ✅ 우수 + 1가지 ⚠️ 양호 |
| **B** | 2가지 이상 ⚠️ 양호, ❌ 없음 |
| **C** | 1가지 ❌ 미흡 있음 |
| **개선 필요** | 2가지 이상 ❌ 미흡 |

## Phase 3에서의 사용법

### Sequential Thinking 9회: 정확성 검증
```
사고 내용:
- 공식 문서와 비교하여 정확성 확인
- deprecated API 사용 여부 점검
- API 시그니처 정확성 확인
- 결론: ✅ 우수 / ⚠️ 양호 / ❌ 미흡
```

### Sequential Thinking 10회: 교육적 구조 검증
```
사고 내용:
- 선택된 패턴이 주제에 적합한지 확인
- 논리적 흐름 점검
- 단계적 진행 확인
- 결론: ✅ 우수 / ⚠️ 양호 / ❌ 미흡
```

### Sequential Thinking 11회: 학습자 친화성 검증
```
사고 내용:
- 전문 용어 설명 여부 확인
- 비유/예시 적절성 점검
- 언어 수준 확인
- 결론: ✅ 우수 / ⚠️ 양호 / ❌ 미흡
```

### Sequential Thinking 12회: 실용성 검증
```
사고 내용:
- 예제의 현실성 확인
- 연습문제 난이도 균형 점검
- 응용 가능성 확인
- 결론: ✅ 우수 / ⚠️ 양호 / ❌ 미흡
```
